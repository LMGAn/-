# 内存池实现与性能分析

## 传统内存管理的缺点
1. **系统调用开销**: 频繁调用malloc/free导致性能下降
2. **内存碎片**: 多次分配释放后，内存碎片化严重
3. **不可预测性**: 动态内存分配可能导致内存不足或延迟

## 内存池的优势
1. **预分配机制**: 一次性分配大块内存，避免系统调用
2. **固定大小元素**: 避免内存碎片
3. **快速回收复用**: 通过空闲链表(Free List)实现O(1)时间的回收操作

## 设计思想
1. **内存块动态扩展**: 初始分配少量块，按需翻倍扩容(blocksUsed <<= 1)
2. **空闲链表复用**: 通过链表管理已释放的内存块，实现高效重用
3. **线程安全**: 使用互斥锁保证多线程环境下的安全性

## 核心数据结构
```c
typedef struct {
    uint32_t elementSize;   // 每个元素的大小（字节）
    uint32_t blockSize;     // 每个内存块能容纳的元素数量
    uint32_t used;          // 当前块中已使用的元素数量
    uint32_t block;         // 当前正在使用的块索引
    uint32_t blocksUsed;    // 已分配的块数组大小（容量）
    uint8_t **blocks;       // 指向块指针数组的指针（二维数组）
    poolFreed* freed;       // 空闲链表头指针
    pthread_mutex_t mutex;  // 线程同步互斥锁
} pool;
```

## 分配策略优先级
1. **优先从空闲链表获取**（最快，O(1)）
2. **当前块内分配未使用空间**
3. **块切换和动态扩容**

## 扩展方向

### 多尺寸内存池
- **需求**: 支持不同大小的对象分配
- **方案**: 为每个尺寸维护独立的内存池（类似SLAB分配器）

### 核心思路
1. **分桶**: 把零散的对象尺寸归并为2的幂级别尺寸桶
2. **路由**: 统一分配/释放接口，内部自动路由到对应尺寸桶
3. **兼容**: 复用单尺寸池的所有特性
4. **边界**: 大对象直接使用系统调用

这个设计是工业级内存池的标准实现，既保留了单尺寸池的高效性，又解决了多尺寸对象的适配问题。
